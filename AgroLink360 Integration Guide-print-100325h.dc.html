1. System overview

Every screen in the wireframe maps to REST endpoints served by the Node/Express API (:4000). The API persists to PostgreSQL/PostGIS and delegates all predictions to the Python AI service (:8000); if the AI service is unreachable, the API answers from built-in heuristics, so the frontend never blocks on the ML stack.

Frontend (React / this wireframe)
  │  fetch + JWT bearer token
  ▼
Node/Express API :4000 ── auth · produce · orders · logistics · payments
  │         │
  ▼         ▼
PostgreSQL+PostGIS   Python AI service :8000 (price · harvest · freshness · spoilage)
                         + MTN MoMo Collection API (payments, escrow)

Local bring-up: cd backend && docker compose up — migrations and demo seed run automatically. Demo logins are in backend/README.md.

2. Authentication

Phone-first auth (no email). Register/login returns a JWT; send it as Authorization: Bearer <token> on every protected call. Role guards are enforced server-side (farmer / buyer / transporter / admin).

Wireframe screenEndpointNotes
Sign up (steps 2–3)POST /api/auth/registerbody: role, name, phone, password (PIN), community. OTP verify = SMS gateway step before this call.
Log inPOST /api/auth/login{ phone, password } → { user, token }
Session restoreGET /api/auth/mecall on app load if a token is stored

4. The prediction model — how data flows in and out

ModelAI service routeTraining data sourceSurfaces in UI as
Dynamic price predictionPOST /priceprice_history table (seeded; grows with every order)harvest card projected price · "below market avg" hint
Harvest windowPOST /harvestplanting dates from listings + agronomic baselines"best window Jul 8–15" card
Freshness estimationPOST /freshnessharvest_date vs shelf-life table per cropgreen bars, "92% fresh" tags, marketplace ranking
Spoilage riskPOST /spoilage-riskcrop shelf-life × delivery ETA (+ weather, phase 2)"high" priority tags on transport requests, farmer alerts
Route consolidation(in Node) /api/logistics/optimizeopen deliveries grouped by pickup communityshared-delivery banner + savings figure

The current models are linear-regression baselines trained at service startup on synthetic seasonal data — enough for the pilot demo. The upgrade path is marked in ai-service/main.py: swap the _train_price_models() fit for XGBoost trained on real price_history rows once ~3 months of transactions accumulate. No frontend change needed — the response contract stays identical.

6. Integration order (recommended)

  1. Auth — wire signup/login to /api/auth, store JWT, restore session on load.
  2. Marketplace read path — GET /api/produce with search/category params; pass browser geolocation for distance ranking.
  3. Farmer listings — POST /api/produce from the modal; render returned freshness_score.
  4. Order + payment loop — POST /api/orders → /api/payments/momo → poll status (sandbox simulates success).
  5. Logistics — requests list, accept, optimize, status updates.
  6. AI cards — harvest/price/spoilage widgets (pure GETs, independent of the above).
  7. Phase 2 — ratings table, live GPS websocket, SMS/USSD gateway (Hubtel/Arkesel), MoMo production credentials + webhook signature verification.
